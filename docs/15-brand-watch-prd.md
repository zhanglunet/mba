# 15 — 品牌舆情监控(Brand Watch)需求分析 PRD

> Status: **需求分析稿(未实现)** · Last verified: 2026-07-12
> 定位:为首页品牌监控台的每个品牌加一层**连续的舆情与信号监控**,作为版本化审计之间的
> "眼睛" 和 EVOLUTION 重审的触发器。本文是需求分析,不含实现;实现拆解归 docs/11 roadmap。

---

## 1. 背景与动机

### 1.1 问题:两次审计之间,监控台是盲的

MBA 的核心产物是**版本化的时点审计**(v1、v2……),EVOLUTION 模式靠人工触发。
这带来一个结构性盲区——**两个版本之间发生的一切,监控台全程无感知**。

真实案例(美团,本仓库 `published/reports/meituan/`):

- v1(2026-06-28)锚在 2023 旧口径,给出 **7.76**(当时审计史最高分);
- v1 与 v2 之间:FY2025 年报由盈转亏、淘宝闪购份额反超、¥7.46 亿食安罚单——这些
  信号在公开渠道陆续出现,但监控台上美团卡片纹丝不动;
- 直到 v2(2026-07-11)人工重审,分数才一次性跌到 **6.72(−1.04)**。

如果有舆情监控:年报发布当天就该在美团卡片上挂出 P0 信号,重审建议自动浮现,
而不是等人想起来。**"监控台"三个字要名副其实,就必须补上这层连续信号。**

### 1.2 与现有体系的关系(先划清边界)

| 已有机制 | 性质 | 舆情监控与它的关系 |
|---|---|---|
| 研究维度 7 `Reception & sentiment`(`references/dimensions.md`) | **时点快照**:审计当刻的舆情横截面 | 舆情监控是**连续时间线**;审计时,维度 7 直接消费监控积累的事件流,不再从零抓 |
| EVOLUTION 模式(docs/12) | 人工触发的增量重审 | 舆情监控是它的**自动触发器**:信号累积过阈值 → 建议重审 |
| MCP `subscribe_brand` / `trigger_evolution`(docs/13) | 订阅与触发的管道已存在 | 舆情监控是**往管道里灌水的水源**;复用,不新造轮子 |
| 评委打分(5 镜头) | 分数的唯一来源 | **舆情监控绝不产生分数**(见 §6 反捏造边界) |

一句话:审计是"体检报告",舆情监控是"日常心率带"。心率带不诊断,但决定你什么时候该去体检。

## 2. 目标用户与场景

沿用 docs/01 的用户画像,场景收窄为三个:

1. **投资人 / FA**:订阅了 5 个品牌,希望"有大事时被叫醒",而不是每周手动重跑审计。
   典型问句:"美团这个月出什么事了?要不要重审?"
2. **品牌方 self-audit**:想知道自家品牌在招投标、监管、社媒上的信号动向,
   以及"哪些信号会在下次审计里伤到哪个镜头的分"。
3. **MBA 自身(Lead agent)**:EVOLUTION 重审时,直接拿事件流作 delta 调研的输入,
   省掉"从零扫全网"的重复劳动——**监控数据是审计的预制菜**。

---

## 3. 设计原则(三条,全部继承自项目立身之本)

1. **硬信号优先**。中标公告、行政处罚、年报数字是"硬信号"(可逐字核查的事实);
   社媒情绪是"软信号"(方向靠模型判断)。硬信号权重恒高于软信号,
   分级体系(§5)按"事实可核查程度"而非"声量"排序。
2. **每条事件可溯源,否则不存在**(反捏造)。事件必须带:原文 URL + 逐字引用 +
   抓取时间戳。情感方向是模型判断,必须标注 `model-judged`,与事实字段物理分列。
   做不到溯源的信号,宁可不入库——与张兰"达不到诚实留 seed"同一条纪律。
3. **分品牌适用,不硬凑维度**。招投标对亚信是生命线,对爱马仕毫无意义;
   每个品牌只开适用的维度(§4.3 适用性矩阵),空维度诚实留空,不填充噪音。

## 4. 舆情监控维度(本 PRD 的核心)

### 4.1 九个监控维度总览

| # | 维度 | 信号性质 | 一句话 | 映射到 5 镜头(主) |
|---|---|---|---|---|
| W1 | 媒体报道 | 软+硬 | 财经/行业/官方媒体的报道与调查 | Signal · Identity |
| W2 | 社交与社区声量 | 软 | 微博/知乎/小红书/X/Reddit 的大众情绪与热点 | Identity · Category |
| W3 | **招投标与采购** | **硬** | 中标/落标/废标/黑名单/合同额——B2B/B2G 的真金白银 | **Leverage · Signal** |
| W4 | 监管、司法与合规 | 硬 | 处罚、问询函、诉讼、召回、禁令 | Signal · Leverage |
| W5 | 资本市场信号 | 硬 | 财报、股价异动、评级、减持、做空报告 | Signal |
| W6 | 产品口碑与投诉 | 软+硬 | 电商/应用商店评分、黑猫投诉、质量事件 | Signal · Identity |
| W7 | 人事与组织 | 硬 | 高管变动、裁员/扩张、创始人动向 | Origin · Leverage |
| W8 | 技术与知识产权 | 硬 | 专利、开源动作、CVE、论文、发射/交付节奏 | Leverage · Category |
| W9 | 供应链与生态伙伴 | 硬 | 合作公告、断供、渠道进退、大客户流失 | Leverage |

> 为什么是"9 维"而不是复用审计的 9 研究维度:审计维度回答"品牌**是**什么",
> 监控维度回答"品牌**发生**了什么"。两套坐标经 `lens_map` 在 5 镜头处汇合。

### 4.2 深潜:W3 招投标与采购(为什么它是硬信号之王)

#### 4.2.1 为什么单列且置顶

对 B2B/B2G 品牌,**招投标结果是唯一"用钱投票"的公开信号**:

- 媒体可以被公关,社媒可以被水军,但**中标公告是甲方拿真预算做出的选择**,
  且依法必须公示——它同时具备"硬信号"与"强制公开"两个属性;
- 它直接映射 MBA 最难核实的两个镜头:**Leverage**(护城河是否还锁得住客户)
  和 **Signal**(营收结构的前瞻指标——中标是 6-18 个月后营收的先行量);
- 本仓库已发布的 13 个品牌里,**6 个的命脉就在招投标**:
  亚信(三大运营商集采)、奇安信(政务采购)、垣信(国家项目)、
  联想(政企 PC 集采)、橙仕(邮政/物流采购)、SpaceX(NASA/DoD 合同)。
  美团审计 v2 的 ¥7.46 亿罚单是 W4,但淘宝闪购份额反超本质上也是"客户用钱投票"的同类信号。

#### 4.2.2 信号分类学(taxonomy)

| 信号类型 | 方向 | 强度基线 | 品牌语义(映射) |
|---|---|---|---|
| **中标**(金额、份额) | pos | 金额 ≥ 年营收 5% → P1,否则 P2 | Leverage:客户锁定仍有效;Signal:未来营收先行量 |
| **落标**(尤其被竞对拿走) | neg | 核心客户落标 → P1 | Leverage:护城河被凿;Category:竞对在重定义供应商格局 |
| **废标 / 流标** | neutral→neg | P2 | Signal:需求侧犹豫或采购规则变化 |
| **独家中标 vs 竞争性中标** | 结构信号 | P2 | 独家=锁定深(Leverage↑)但依赖单一客户(脆弱性↑),双面标注 |
| **中标价格趋势**(同类标的连续降价) | neg | 连续 2 次显著降价 → P1 | Signal:定价权流失——比营收下滑更早暴露 |
| **续约 / 复购** | pos | P2 | Leverage:存量锁定的直接证据 |
| **被列入采购黑名单 / 处罚性禁入** | neg | **P0** | Leverage 结构性受损;联动 W4 |
| **竞对重大中标**(本品牌未参与) | 参照系 | P3(仅入库) | Category:品类格局变动的背景板 |

#### 4.2.3 数据源清单(可达性按坑 #1 纪律诚实标注)

> 标注约定:✅=本仓库已验证可 curl;🔍=公开但**接入前需逐源验证**(反爬/JS 墙未测);
> 💰=商业付费库,只作线索不作一手。**接入 SOP:每个源先跑一次真实 curl 验证再标 ✅,
> 不凭印象标注**——这条与 CLAUDE.md 坑 #1 是同一条纪律。

**中国(政府与公共采购)**
- 中国政府采购网(ccgp.gov.cn)— 中央与地方政府采购公告的法定发布平台 🔍
- 全国公共资源交易平台(ggzy.gov.cn)及各省分平台 🔍
- 军队采购网(涉密项目不公示,只有公开部分)🔍

**中国(行业与央企集采——亚信/奇安信/垣信的主战场)**
- 中国移动采购与招标网(b2b.10086.cn)🔍 · 中国电信阳光采购网 🔍 · 中国联通采购网 🔍
  ——**运营商集采公告是亚信 BSS 命脉的直接可观测量**,单此三源即覆盖其核心 W3
- 国家电网 ECP、中石油/中石化等央企电子招标平台 🔍
- 上市公司中标公告:巨潮资讯网(cninfo)/ 港交所披露易 / 上交所·深交所公告 🔍

**美国(SpaceX 等)**

- NASA / DoD / Space Force 新闻稿(commercial crew、NSSL 等大单均有官方公告)🔍

**聚合与商业库**
- 千里马、剑鱼标讯、企查查中标模块等 💰 — 只作发现线索,入库必须回溯到法定公示原文

#### 4.2.4 已发布品牌上的落地示例(用真实案例说明信号→镜头链路)

- **亚信**:三大运营商年度集采中,亚信科技在 BSS 标段的**份额与单价**逐年可比。
  若出现"份额保住但单价连降"——这正是 v1 报告里"BSS 收缩(−19.1%)"的**前 6 个月先行信号**,
  映射 Signal(定价权)与 Leverage(锁定依旧但变现变薄),重审时评委可精确降 Signal 而不动 Leverage。
- **垣信**:它现阶段 pre-revenue,W3 反向使用——**垣信作为甲方的采购公告**
  (火箭发射服务、卫星载荷)反映组网节奏是否兑现,直接回应 v1 报告最大质疑
  "频段不等于在轨"(马斯克△ 的 Δ5 分歧点)。
- **SpaceX**:NSSL Phase 3、NASA 合同的授予/份额变化,是 Starlink 之外
  发射业务 Leverage 的官方可核查信号(SAM.gov 有法定记录)。

## 5. 事件模型与分级

### 5.1 事件 schema(单一事实源:`watch/<slug>/events.yaml`)

```yaml
- id: 2026-07-12-asiainfo-001
  date: 2026-07-12                  # 事件公开日(非抓取日)
  dim: W3                           # W1..W9
  severity: P1                      # P0..P3,见 5.2
  direction: neg                    # pos / neg / neutral / mixed —— 模型判断
  direction_by: model-judged        # 恒为 model-judged,物理分列于事实字段
  title: 某运营商 2026 年 BSS 集采结果公示,亚信份额下降
  quote: "……(公告原文逐字引用,≤100 字)……"   # 必填,写入前 grep -F 核查
  url: https://…                    # 法定公示 / 一手源优先
  fetched_at: 2026-07-12T05:00:00Z
  lens_map: [leverage, signal]      # 映射到 5 镜头(供重审消费)
  note: 单价较 2025 年下降约 x%(据公告金额与标段数推算,estimate)
  # ── 舆情驾驶舱扩展字段(可选,docs/20)——均为标签/判断,不改硬字段口径 ──
  related_persons: ["田溯宁"]        # 关联人物(管理层/关键人物);须为源文本里真实出现的人名
  source_type: finance              # official|media|finance|social|investor_community|search|regulator
  suggested_action: "统一 IR 口径"  # 建议动作(结构化);note 仍留作长文说明
  alert_tier: L2                    # L1/L2/L3 预警层级覆写(缺省由 severity+触发规则推导,见 notify_feishu)
```

**硬字段**(date/quote/url/fetched_at)与**判断字段**(direction/severity/lens_map)
分列。判断字段允许重审时被评委推翻;硬字段错了就是事故。

**舆情驾驶舱扩展字段(4 个可选,docs/20)**:`related_persons`(关联人物,须取自源文本真实人名,
不杜撰)、`source_type`(来源类型枚举)、`suggested_action`(结构化建议动作)、`alert_tier`
(预警层级覆写)。全部向后兼容:旧事件不填不影响;校验由 `validate_watch.py` 的枚举 gate 兜底。

### 5.2 分级(按"对下次审计分数的预期影响"定义,不按声量)

| 级别 | 定义 | 判例 | 动作 |
|---|---|---|---|
| **P0** | 单事件即足以改变某镜头评分 | 监管处罚落地、采购黑名单、做空报告、重大安全事故、创始人出走 | 卡片即时挂标,**建议立即 EVOLUTION** |
| **P1** | 与同类事件累积后大概率改分 | ≥年营收 5% 的中标/落标、财报大幅偏离、定价连降、重大断供 | 卡片计数,入触发窗口 |
| **P2** | 显著但单独不改分 | 常规中标/落标、高管变动、显著舆论事件、评级调整 | 入库+周报汇总 |
| **P3** | 背景噪音,只作语境 | 常规报道、一般社交讨论、竞对动态 | 只入库,**不上卡片** |

### 5.3 触发规则(舆情 → EVOLUTION 建议)

滚动 30 天窗口内:`P0 ≥ 1` **或** `P1 ≥ 3` **或** 加权计数(P0=4, P1=2, P2=0.5)≥ 6
→ 该品牌卡片挂 **「建议重审」** 徽章,并经 MCP `subscribe_brand` 订阅链路下发建议。

> **2026-07-12 校准**:初版阈值为 `P1≥2` / 加权 ≥5,在首批 5 次重审回测中精确率仅
> 1/5(docs/16 §8.3)——收紧为 `P1≥3` / 加权 ≥6 后压掉两个最差误报且不丢真阳性
> (回测 1/3)。随样本积累可继续调参,调参对象只能是**触发门槛**,不能是评分。

**边界(反捏造的延伸,不可妥协)**:舆情监控**只建议、不改分**。
监控台上的分数永远来自最近一次评委重审;任何"舆情实时分""情绪指数打分"
都是把模型判断伪装成审计结论,违反项目立身之本,明确列为**非目标**(§8)。

## 6. 产品形态

### 6.1 首页监控台(增量,不重构)

- 品牌卡新增**舆情徽章**:如 `⚡ P0×1` / `📋 P1×2`(30 天窗口计数,P3 永不上卡);
- 触发规则命中时显示 `建议重审` 徽章(样式对齐现有 `badge-mode`);
- 数据经 `build_home_cards.py` 从 `watch/*/events.yaml` 生成——**沿用单一真源 + 漂移
  gate 模式**(坑 #2 同款纪律,徽章区进 REPORTS 生成标记内)。

### 6.2 品牌舆情时间线(新二级页)

`/watch/<slug>/`:按时间倒序的事件流,P0/P1 高亮,每条事件直链原文;
顶部按 W1-W9 维度过滤。报告页(`/reports/<slug>/`)尾部加"自上一版以来的信号"摘要区,
链到时间线——把"两版之间发生了什么"补进报告叙事。

### 6.3 CLI 与 MCP

- `/mba <brand> --watch`:单次舆情扫描(M1 的形态),按适用性矩阵扫开启的维度,
  产出/更新 `events.yaml`,并打印触发规则评估结果;
- EVOLUTION 流程改造:Phase 2 delta 调研**先消费 events.yaml 再补扫**,
  维度 7 的时点快照与连续时间线在此汇合;
- MCP 新增 2 工具:`get_watch_events(brand, since?)` / `record_watch_event(brand, event)`;
  触发建议复用既有 `subscribe_brand` / `trigger_evolution`,不加新管道。

## 7. MVP 分期

| 阶段 | 内容 | 验收 |
|---|---|---|
| **M1 半自动**(先跑通"信号→审计"闭环) | ① 定 schema + 适用性矩阵落库;② `--watch` 单次扫描(复用现有 curl + WebSearch 能力);③ 试点 **3 个 W3 核心品牌:亚信、奇安信、垣信**(招投标信号最硬、源最集中);④ 下一次 EVOLUTION 消费事件流 | 试点品牌各产出 ≥10 条可溯源事件;其中 ≥1 次重审的 delta 调研直接引用事件流 |
| **M2 定期采集** | CCR Routines / cron 每周扫描;首页徽章上线(含漂移 gate);/watch 时间线页 | 13 品牌适用维度覆盖 ≥80%;徽章与 events.yaml 零漂移 |
| **M3 告警与联动** | 触发规则自动评估;P0 推送通知;MCP 订阅链路下发重审建议 | 触发建议精确率 ≥60%(见 §9) |

M1 刻意选 W3 试点而非 W2(社交):招投标源结构化程度最高、反爬最弱、
且完全避开"情感判断准不准"的争议——先用硬信号证明闭环,再扩软信号。

## 8. 非目标(Explicitly out of scope)

1. **不做"舆情实时评分"**——任何形态的自动打分都不做(见 §5.3 边界);
2. **不做全网监控 SaaS**——只监控本站已发布品牌,维度按矩阵收敛;
3. **不抓不可达源**——知乎/微博/澎湃等反爬源(坑 #1)在打通合规通道(如 Wuying)前,
   相应维度诚实标"源不可达",不用二手转述冒充;
4. **不存全文**——只存 ≤100 字逐字引用 + URL + 快照时间戳,版权与存储双重考量;
5. **v1 不做情感模型微调**——direction 就是 LLM 判断 + 显式标注,不假装客观。

## 9. 成功指标

| 指标 | 目标 | 度量 |
|---|---|---|
| 事件可溯源率 | **100%(硬性)** | 每条事件 URL 可访问 + quote 逐字命中原文(抽检) |
| 信号→审计转化率 | ≥70% | P0/P1 事件被下一版审计报告引用的比例 |
| 触发建议精确率 | ≥60% | 「建议重审」后,重审总分变动 ≥0.3 的比例 |
| 先行性 | 中位 ≥14 天 | P0/P1 事件公开日早于对应重审日的天数 |
| 覆盖 | 矩阵 ● 维度 100% 接入 | 按 §4.3 核心维度计 |

## 10. 风险与开放问题

| 风险 | 应对 |
|---|---|
| 数据源反爬 / 失效(坑 #1 的日常化) | 接入前逐源 curl 验证;每源记录最近成功抓取时间,连续失败 → 维度标"源退化"而非静默留空 |
| 招投标数据碎片化(法定平台分散、聚合库收费) | M1 只接"运营商集采 + 政府采购网 + 上市公司公告"三类高置信源;聚合库仅作发现线索 |
| 软信号方向误判 | direction 恒标 model-judged;P0/P1 软信号需双源交叉才入级,单源降 P2 |
| 噪音淹没(P3 洪水) | P3 永不上卡片;入库但仅供重审时作语境 |
| 版权 / 合规 | 只引 ≤100 字 + 链接原文;不转载全文;招投标公示信息为法定公开数据 |
| 开放问题 | ① events.yaml 放 `watch/` 还是 `published/reports/<slug>/watch/`(与发布白名单的关系)?② P0 通知走什么渠道(GitHub issue / 邮件 / MCP 推送)?③ 竞对事件(P3 参照系)要不要独立品牌页?——留给实现阶段决策 |
