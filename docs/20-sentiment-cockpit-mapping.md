## 1. 需求背景(通用模式)

多主体、上市、B2B/B2G 型企业,普遍有一类舆情监测诉求:

- **主体多、渠道多、口径杂**:集团 + 子品牌 + 管理层关键人物,分布在不同媒体矩阵。
- **外采舆情公司的短板**:不理解内部业务与品牌架构 → 信息偏差大、去噪难、人工成本高。
- **产出形态落后**:多为人工日报/周报,看板不可定制,缺小时级更新与管理层视角。
- **目标**:从被动通报 → **主动预警与主动管理**,并反哺品牌传播选题。

## 2. 目标用户与场景

| 用户 | 场景 |
|---|---|
| 市场公关 | 守住品牌声誉底线,及时捕捉负面/谣言/投诉;危机公关处置 |
| 投资者关系(IR) | 资本市场信号、投资者情绪、关联交易讨论 |
| 管理层 | 掌握"发生了什么、影响多大、是否启动危机公关";高层预警 |

## 3. 功能需求(结构化)

### 3.1 监测主体体系(3 类)
公司主体(集团 + 子品牌) · 管理层与关键人物 · 业务与风险主题

### 3.2 数据来源(多渠道)
科技媒体 · 财经证券媒体 · 搜索引擎 · 社交平台(微信/微博)· 投资社区/炒股软件(雪球/东财/同花顺)

### 3.3 自动标注(每条 7 标签)
风险等级 · 舆情倾向 · 来源类型 · 关联主体 · 关联人物 · 风险主题 · 建议动作

### 3.4 系统板块(4 区)
1. **管理层摘要区**:总样本数、重点风险、正面传播、投资社区动态 + 发布时间分布(是否集中爆发)
2. **风险判断区**:风险主题归因(传播/关联交易/投资者讨论/资本市场/技术安全/业财)+ 重点风险卡片 + 建议动作
3. **投资社区专区**:雪球/东财/同花顺;观察讨论热度随公告/股价/业绩/关联交易的变化
4. **全量信息表**:按日期/主体/主题/等级/来源筛选,追溯原文链接

### 3.5 监控频次
7×24 小时持续监控;小时级更新(优于日/周报)

### 3.6 分层推送 / 预警
第一层 日常监测(部门知晓)· 第二层 专项协同(协同部门)· 第三层 高层预警(条件触发)

## 4. 舆情驾驶舱的通用工程难点 / 硬约束

> 任何自建舆情方案都绕不开的地板,PRD 必须显式承接:

- **C1 数据源宽度**:招投标(中标/串标/被处罚)、私域/半私域、甚至内网——很多舆情先从内部冒烟、外溢才爆。
- **C2 真舆情 vs 投毒**:需去伪、防误报/漏报、防"警报疲劳"。
- **C3 舆情是业务环、非展示**:核实来源真实(防误报)→ 部门处理 → 公告/回应 → 沉淀 → 多部门联动;是长链条,不是给领导看到就结束。
- **C4 高价值网站反爬**:高价值站点对 AI/爬虫反制很深,工程实现难。
- **C5 运营配套**:不只是技术问题,需要专门运营人力配合关键词/事件维护。
- **C6 自建 ≠ 立刻替代**:短期内难替代成熟舆情公司,宜并行/合作。

## 6. MBA 的增量价值(通用舆情聚合方案没有的)

1. **人物评委研判,而非单纯聚合** —— 不止"打标签 + 建议动作",能让一条舆情经**具名人物视角**给出结构化研判,直指品牌影响力的 5 镜头。
2. **版本化、可复盘** —— 舆情非一次性告警:示例 `asiainfo` 有 v1(5.93)→ v2(5.77)的 EVOLUTION delta,判断有据、可回放。正面回应 C3。
3. **反投毒 = 机器门禁** —— "投毒/误报"在 MBA 是**已上线 CI 硬 gate**:写入引用必须源站逐字命中,造假即红。多数外采与自建方案都没有这层地板。
4. **飞书触达已打通** —— 分层预警(日常/专项/高层)可直接复用 `notify-feishu.yml`(P0/P1 + 建议重审 + 评分变动),门槛可调,缓解 C2 的警报疲劳。

## 7. 落地建议

**可立刻做**:主体聚合为一个品牌;主用 **W1/W2/W3/W4/W5** 五维(覆盖数据源诉求 + 补招投标);7 标签直接落到事件 schema;分层预警走飞书;看板走监控台 + 私有星图。

**需适配 / 排期**:关联人物标注、投资社区取数适配;处置闭环(C3)建议**对接现有工单/协同系统**,MBA 只做"建议重审 + 版本化留痕",不越界做 CRM。

**诚实边界(不夸大)**:
- ⚠️ MBA **不做自动化反爬**、不覆盖私域/内网、非小时级实时(C1/C4 硬工程难点,需人工核对或采买数据源)。
- ⚠️ 工具**补强而非替代**成熟舆情公司;成熟前建议与外部数据源并行。
- 任何降本增效数字须标注来源、独立验证,不作为 MBA 承诺。
